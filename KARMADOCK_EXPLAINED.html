How the predicted poses are produced

run_infer.sh runs KarmaDock's ligand_docking.py with --docking --scoring --correct --random_seed 2023, docking each complex 3 times (3 random ligand starts = "repeats"). Each repeat yields one pose plus a confidence MDN score. KarmaDock writes, per repeat re ∈ {0,1,2}:

# the score table — one row per complex
kd_out/<re>.csv   columns: pdb_id, score, RMSD, FF_RMSD, Aligned_RMSD
# the pose, in three correction variants
kd_out/<re>/<id>_pred_{uncorrected|ff_corrected|align_corrected}.sdf

How we guarantee "ranked best-first" the key answer

Higher MDN score = better pose. Verified on the real cluster CSV (kd_out/p1_baseline/proto_test/0.csv):

pdb_id,score,RMSD,FF_RMSD,Aligned_RMSD
5s7b_LU8_A_501,68.25,1.95,1.87,2.82   # high score → low RMSD (good)
5s7d_LU8_A_508,13.36,9.13,9.23,9.71   # low score  → high RMSD (bad)

convert_karmadock_to_seminar.py collects the 3 repeat-scores per complex, sorts them descending by MDN score, then writes the poses into one multi-conformer <id>_pred.sdf in that order — best pose first:

# convert_karmadock_to_seminar.py (lines 58–59)
ranked = sorted(scores[cid], key=lambda x: x["score"], reverse=True)  # higher MDN score first

Verified output (results/p1_baseline/proto_test/3ix2_AC2_A_302_pred.sdf): 3 poses, each tagged KarmaDock_Score / KarmaDock_Repeat / KarmaDock_Mode. This satisfies the tutor's spec — results/README.md: "Poses should be ordered by confidence/score (best pose first)."

Why it matters. The evaluator scores the top pose. If the file weren't sorted best-first, a genuinely good prediction would be read as bad. This one sort is the correctness hinge of the whole results step.

How PoseBusters is prepared

Notebook §3e runs PoseBusters directly on the exported (already-ranked) poses:

from posebusters import PoseBusters
buster = PoseBusters(config='redock')   # predicted pose vs crystal ligand vs protein

It checks the top pose of each complex and reports PB-Valid = fraction passing all physical checks (P1 13.3% / P3 6.7% / P2 0%, dominated by protein-clash failures). PoseBusters' own rmsd ≤ 2 Å and the *_loaded I/O checks are reported separately, not folded into PB-Valid.

Two more caveats to carry into the review.
  • Submitted variant: run_infer.sh exports --mode align_corrected, but we ship uncorrected (best on proto_test) — confirm the files in results/proto_test/ carry KarmaDock_Mode = uncorrected.
  • Two different RMSDs: PoseBusters' symmetry-corrected RMSD ≠ KarmaDock's internal RMSD; the tutor's official number uses symmetry-corrected, so it may differ from the notebook's figures.
  • results/README.md lists three datasets (proto_test, posebusters_filtered, full_test) — only proto_test is in scope for the prototype.

1. What is this project?

The seminar (Benchmarking DL-based Docking Tools, Saarland University) gives each team one deep-learning docking tool. Our tool is KarmaDock. Docking means: given a protein (a big biological molecule, like a lock) and a small ligand (a drug-like molecule, like a key), predict how and where the ligand fits into the protein's pocket — its 3D pose.

Our job has three parts:

1
Run the tool (inference) on a shared dataset and measure how good its predicted poses are.
2
Re-train the tool ourselves on the seminar's data — to compare a from-scratch model and a fine-tuned model against the released one.
3
Analyze & present the results, comparing them to the numbers in the original paper.

The twist: KarmaDock's public code only ships the part that runs a trained model (inference) plus the pretrained weights. It does not include a training script. So the heart of our contribution is train.py — a training loop we wrote ourselves around KarmaDock's model. That is the file the tutor is most likely to ask about, and most of this page builds up to explaining it.

2. The big picture

Three "pipelines" all share the same plumbing; they differ only in which weights the model uses:

PipelineWhere the weights come fromQuestion it answers
P1 — baselinethe authors' released weights (karmadock_screening.pkl)How good is KarmaDock "out of the box"?
P2 — from scratchrandom start, trained only on seminar dataCan we reproduce its training from zero?
P3 — fine-tunereleased weights, then trained a bit more on seminar dataDoes extra training on this data help?

The flow is the same for all three:

raw protein + ligand files→ preprocess (build graphs)→ train (P2/P3 only)→ dock (inference)→ predicted poses→ evaluate (RMSD)

We run everything on the university HPC cluster using HTCondor (a job scheduler) inside a Docker container (a frozen, reproducible software environment). More on those file types next.

3. File formats glossary

Every file type you will see in the project, what it holds, and why it exists:

.pdb input · protein
Protein structure. A plain-text file listing every atom of the protein and its 3D coordinates (x, y, z), plus which amino-acid and chain it belongs to. Comes from the Protein Data Bank. Example: 3ix2_AC2_A_302_protein_refined.pdb.
.sdf input/output · ligand
Small molecule structure. Lists the ligand's atoms, their 3D coordinates, and the bonds between them. A single .sdf can hold several poses ("conformers") of the same molecule stacked together. The crystal (true) ligand is <id>_ligand_refined.sdf; our prediction is <id>_pred.sdf.
.csv index
The dataset index. A spreadsheet (comma-separated) with one row per complex, naming its ligand file, protein file, the measured binding affinity, year, and PDB id. Example columns: ligand_file_name, protein_file_name, Year, Log Binding Affinity, …. Our code reads this to know which complexes exist.
.dgl preprocessed · graph
The graph the model actually eats. KarmaDock turns each protein+ligand into a graph (atoms = nodes, bonds/contacts = edges) and saves it with the DGL library. One file per complex: 3ix2_AC2_A_302.dgl. Building these is the slow "preprocessing" step; afterwards training/inference just loads them.
.pkl model weights
A trained model checkpoint ("pickle"). Holds the millions of learned numbers (weights) of the network. karmadock_screening.pkl = the authors' weights; karmadock_team002.pkl = our best trained weights.
last.pt resume file
A full training snapshot (PyTorch). Unlike the .pkl (best weights only), this also stores the optimizer state, current epoch, and best score — so a job killed mid-training can resume exactly where it stopped.
train_log.csv training history
One row per epoch of training: epoch number, train loss, validation loss, etc. This is what the notebook plots as "training curves".
.py python code
Python programs. train.py (our trainer), the converters, the adapters. The real logic lives here.
.sh bash wrapper
Shell scripts — short "recipe" files that call the Python programs with the right arguments in the right order (e.g. preprocess.sh, run_infer.sh). They save you from typing long commands by hand.
.sub cluster job
HTCondor submit files. They tell the cluster what to run (which .sh), with how many CPUs/GPUs/memory, which Docker image, and where to write logs. You "submit" one with condor_submit file.sub.
.ipynb notebook
Jupyter notebook — code + text + plots in one document. Our results and data-exploration analyses live here.
Dockerfile environment recipe
The build recipe for the Docker image — it installs Python, PyTorch, RDKit, KarmaDock, and copies our scripts in, so every cluster run uses the exact same software.
.out / .err / .log run records
Produced by every cluster job: .out = normal printout, .err = errors/warnings, .log = HTCondor's own record (when it started, where it ran).

4. The KarmaDock paper, in plain words

Paper: Zhang, Zhang, Shen, … Hou. “Efficient and accurate large library ligand docking with KarmaDock.” Nature Computational Science 3, 789–804 (2023).
DOI: 10.1038/s43588-023-00511-5 · Code: github.com/schrojunzhang/KarmaDock

Traditional docking programs (AutoDock Vina, Glide, GOLD) search over millions of possible ligand positions and score each with a hand-made formula. That is accurate-ish but slow. KarmaDock replaces the search with a neural network that moves the ligand straight into place, making it about ~100–1000× faster — fast enough to screen huge compound libraries.

The paper describes a three-part model (quoting the abstract, simplified):

PartWhat it doesBeginner analogy
1. EncodersRead the protein and the ligand and turn each into a numeric "understanding" of its shape and chemistry (graph neural networks).Reading the lock and the key and memorizing their features.
2. E(n)-equivariant GNN (with self-attention)Iteratively moves the ligand's atoms to dock it into the pocket, using protein–ligand and internal interactions; then a post-processing step fixes the geometry so it is chemically sensible.Gently rotating & sliding the key until it sits in the lock, then straightening any bent teeth.
3. Mixture Density Network (MDN)Predicts a binding score — how strongly the ligand sticks — from the predicted distances between atoms.Judging how well the key actually turns the lock.

"E(n)-equivariant" is the clever bit: if you rotate or move the whole protein in space, the network's predicted pose rotates/moves with it instead of breaking. That is exactly the property docking needs, and it is why the model can learn geometry efficiently.

5. The two losses (this is the key to train.py)

To train any neural network you need a loss: a single number saying "how wrong are you right now?". Training nudges the weights to make that number smaller. KarmaDock has two jobs, so it has two losses:

① Pose loss — rmsd_loss
How far the predicted ligand atoms are from their true (crystal) positions. Small = the pose is in the right place. This trains Part 2 (the docking GNN).
② Scoring loss — mdn_loss
How badly the MDN's predicted distance distribution matches reality. Small = the binding score is trustworthy. This trains Parts 1 & 3 (encoders + MDN).

The total loss is just a weighted sum, controlled by one knob called pos_r ("position ratio"):

# straight from KarmaDock's own model.forward()
rmsd_loss, mdn_loss = model(data, device, pos_r)
loss = pos_r * rmsd_loss + mdn_loss

This one line is the whole reason we could write a trainer without the authors' training code: the model already returns both losses itself. Our job was to build the loop around it. The pos_r knob is also how we switch between the two training stages (next section):

  • pos_r = 0 → ignore the pose loss → only train scoring.
  • pos_r = 1 → use both → train docking + scoring together.

6. The two-stage training protocol

The paper (Methods, "Training protocol") trains in two stages, and our P2 from-scratch pipeline copies this exactly:

1
Stage 1 — scoring only (pos_r = 0). First teach the encoders + MDN to understand chemistry and judge binding, without yet worrying about moving atoms. Learning rate 1e-3, a little weight-decay (1e-5).
2
Stage 2 — docking + scoring (pos_r = 1). Now also learn to move the ligand into place, starting from Stage-1's best weights. Smaller learning rate 1e-4, no weight-decay.

Why two stages? If you ask the model to dock and score from random weights at once, both signals are noisy and it learns slowly. Warming up the "understanding" first makes the harder docking step much more stable.

Fine-tuning (P3) is just "Stage 2 on its own." Because the released weights already know chemistry, we skip Stage 1 and do a single pos_r = 1 run starting from karmadock_screening.pkl, with a shorter patience so it doesn't overfit.

8. train.py — the deep dive

This is our main artifact (~460 lines). Below it is broken into the blocks it is actually made of, in order, each explained in plain English. The file lives at scripts/train.py.

8.1 The header & the core idea (lines 1–25)

The docstring states the whole trick: KarmaDock ships no trainer, but its model(...) returns the two losses, so we wrap them in a loop. It also names the two modes — --init_model "" (from scratch) vs --init_model karmadock_screening.pkl (fine-tune).

8.2 Imports (lines 27–44)

import torch, torch.nn, torch.distributed # the deep-learning engine + multi-GPU
from utils.fns import set_random_seed, Early_stopper, karmadock_evaluation
from dataset.graph_obj import PDBBindGraphDataset      # loads .dgl graphs
from dataset.dataloader_obj import PassNoneDataLoader   # batches them, skips broken ones
from architecture.KarmaDock_architecture import KarmaDock   # the model itself

We reuse the authors' own classes (model, dataset, dataloader, helper functions) — we did not rewrite the model. We only add the training loop. That is an honest, important point for the tutor.

8.3 Multi-GPU helpers (lines 63–80)

These let one training run spread across several GPUs (DDP = DistributedDataParallel). They are opt-in: with a normal python3 train.py they do nothing, so single-GPU results stay identical.

  • ddp_info() — detects whether we were launched in multi-GPU mode.
  • shard(ids, rank, world) — splits the complex list so each GPU trains on a different slice (ids[rank::world]).
  • all_reduce_sum(...) — adds up numbers across all GPUs so they agree on the validation score.

8.4 The settings — parse_args() (lines 83–129)

Every knob you can turn from the command line. The important ones:

ArgumentMeaning
--csv, --graph_dirwhich complexes to train on, and where their .dgl graphs are
--init_modelempty = from scratch; a .pkl = fine-tune from those weights
--pos_rthe loss knob from §5 (0 = scoring only, 1 = docking+scoring)
--lr, --weight_decayhow fast / how regularized the learning is
--batch_size, --accum_stepseffective batch = batch_size × accum_steps (paper uses 64)
--patiencestop if validation hasn't improved for this many epochs (early stopping)
--val_csvuse a separate validation set (needed for the leak-proof LP-PDBBind split)
--resumecontinue from last.pt if a cluster job was interrupted

8.5 Reading the dataset (lines 132–169)

complex_ids_from_csv() turns each row of the CSV into a complex id by stripping the filename suffix (3ix2_AC2_A_302_ligand_refined.sdf → 3ix2_AC2_A_302). Then split_available() keeps only ids that actually have a .dgl graph (some drop out when RDKit can't read the molecule) and splits them into train/validation deterministically (same random seed → same split every time, so results are reproducible).

8.6 Gradient accumulation, explained (lines 200–213 & 354–383)

The paper trains with an effective batch of 64 complexes, but 64 graphs at once don't fit in GPU memory. The trick: process a small batch (e.g. 4), compute gradients but don't update yet, repeat 16 times adding the gradients up, then update once. 4 × 16 = 64 — same result as a batch of 64, but it fits. 

for data in train_loader:
    rmsd_loss, mdn_loss = model(data, device, args.pos_r)   # forward
    loss = args.pos_r * rmsd_loss + mdn_loss                # §5 total loss
    (loss / args.accum_steps).backward()                    # add to gradient
    if accum % args.accum_steps == 0:                       # every 16 mini-batches…
        nn.utils.clip_grad_norm_(model.parameters(), 10.0)  # cap huge gradients
        optimizer.step(); optimizer.zero_grad()             # …update the weights once

clip_grad_norm_ caps the gradient size so one weird complex can't blow up training.

8.7 The training loop (lines 350–411)

For each epoch (one pass over all training complexes):

1
Set the model to "train" mode and loop over batches, doing the forward → loss → backward → (accumulate) → step shown above.
2
Validate: run the model on the held-out validation complexes without learning, and average the loss. This is the honest "how good are we?" number.
3
Log one line to train_log.csv and (if enabled) to Weights & Biases for live curves.

8.8 Checkpoints & early stopping (lines 413–441)

After each epoch the code:

  • saves last.pt (full snapshot, for resuming),
  • if the validation loss is the best so far, also saves karmadock_team002.pkl (our deliverable),
  • if validation hasn't improved for --patience epochs, stops early — no point overfitting.

This is why a multi-day cluster run survives interruptions: it always has a recent last.pt to resume from, and --resume picks it up automatically.

8.9 Weights & Biases run identity (lines 237–275)

W&B is an online dashboard for training curves. The careful part here is giving each run a unique id (name_timestamp_pid) saved to wandb_run_id.txt, so a re-submitted job creates a new record instead of silently overwriting an old one — a bug we hit and fixed (earlier the id was a hash of the output folder, which collided when a folder was reused).

9. The wrapper scripts (the .sh files)

These are short recipes that call the Python programs with the paper's exact settings, so a cluster job is one clean command.

preprocess.sh

convert_seminar_to_karmadock.py   # 1. seminar files → KarmaDock folder layout
pre_processing.py                 # 2. cut the 12 Å pocket around the ligand (prody)
generate_graph.py                 # 3. build the .dgl graph per complex

Turns raw .sdf/.pdb into the .dgl graphs the model needs. CPU-only, run once per dataset.

run_train_paper.sh — Pipeline 2 (from scratch)

Runs train.py twice, implementing the two-stage protocol from §6:

# Stage 1: scoring only
train.py --init_model "" --pos_r 0 --lr 1e-3 --weight_decay 1e-5 --patience 70 …
# Stage 2: docking+scoring, starting from Stage-1's best
train.py --init_model stage1/karmadock_team002.pkl --pos_r 1 --lr 1e-4 --weight_decay 0 --patience 70 …

A stage.done marker stops Stage 1 from re-running if the whole job restarts.

run_finetune.sh — Pipeline 3 (fine-tune)

A single Stage-2-style run starting from the released weights:

train.py --init_model …/karmadock_screening.pkl --pos_r 1 --lr 1e-4 --weight_decay 0 --patience 30 …

Shorter patience (30 vs 70) because a pretrained model on a small dataset converges fast and could overfit.

run_infer.sh — docking / inference

ligand_docking.py --docking True --scoring True --correct True --batch_size 64 --random_seed 2023
convert_karmadock_to_seminar.py   # export the predicted poses

Uses KarmaDock's own docking code with a model file (released or ours), then exports the poses. --correct True turns on the paper's post-processing (force-field / alignment) so the geometry is chemically clean.

10. The converters (format glue)

KarmaDock and the seminar name their files differently, so two small adapters translate between them.

convert_seminar_to_karmadock.py

Seminar layout (flat folder, <id>_ligand_refined.sdf) → KarmaDock layout (one folder per complex, <id>/<id>_ligand.sdf). It just reads the CSV and copies files into the right shape.

convert_karmadock_to_seminar.py

KarmaDock docks each complex 3 times (3 random starts = "repeats") and writes a score per repeat. This script ranks the repeats by MDN score (higher = better) and writes them into one <id>_pred.sdf best pose first, because the evaluator scores the top pose. It can export three pose variants:

--modeMeaning
uncorrectedthe raw network pose (best on our data → our submission)
ff_correctedforce-field cleaned (most physically realistic, per the paper)
align_correctedalignment-corrected variant
Why best-pose-first matters. The tutor's evaluator looks at the top pose. If the file isn't sorted best-first, a good prediction can look bad. This sorting is a small but important correctness detail.

11. The one-submit driver — run_all_pipelines.sh

For the full-data round we built a single job that does everything for a dataset, so when new data arrives we change one line (the dataset name) and submit once. It chains: preprocess → P1 → (if a train set exists) P2 → P3 → inference for each, then promotes the headline poses to the path the evaluator expects. 

condor_submit all_pipelines.sub        # DATASET = proto  (the only line you edit)

It is idempotent: re-running skips finished steps (existing graphs / existing poses) and resumes training, so an interrupted job continues instead of starting over.

13. If the tutor asks… (talking points)

“Where did your training script come from — did you write it?”
Yes. KarmaDock ships inference + weights only, no trainer. We wrote train.py as a loop around the authors' model, which already returns both losses (rmsd_loss, mdn_loss). We did not modify the model itself — we reuse their classes.
“What is pos_r and why does it change between runs?”
It's the weight on the pose loss. pos_r=0 trains scoring only (Stage 1); pos_r=1 trains docking + scoring (Stage 2). It's how we implement the paper's two-stage protocol from one script.
“How do you fit a batch of 64 on a 16 GB GPU?”
Gradient accumulation: 16 mini-batches of 4, summing gradients, then one optimizer step. Mathematically equal to a batch of 64.
“What happens if a multi-day job is killed?”
Every epoch we save last.pt (weights + optimizer + epoch). With --resume the job reloads it and continues from the next epoch — no work lost.
“What's the difference between the two checkpoint files?”
karmadock_team002.pkl = best weights only (the deliverable / for inference). last.pt = full snapshot for resuming. Both are written during training.
“Why three pose files per complex and why sorted?”
KarmaDock docks 3 times from different random starts. We rank those repeats by MDN score and write best-first, because the evaluator scores the top pose.
“How is this measured?”
RMSD between the predicted top pose and the crystal ligand; "success" = RMSD < 2 Å. Reported as a success rate over the test set, compared to the paper's numbers.
